# 組織を生命として捉える系譜分析 実行ロードマップ

作成日: 2026-05-02  
位置づけ: 第1波調査 `research/A`〜`F`, `codex1`〜`codex3` を踏まえた v0.0 から v1.0 までの実行計画。期間・工数・コストは未確定であり、本文中の期間は仮置きである。

## プロジェクト概要 — 目的・成果物・原則

目的は、組織を単なる法人・企業・制度分類ではなく、誕生、境界維持、資源動員、知識継承、関係形成、変容、消滅を行う歴史的な「作動単位」として比較し、古代・中世・近代・現代・デジタル組織を横断する系譜を描くことである。

主要成果物:

- 理論枠組み: 個体群生態学、オートポイエーシス、サイバネティクス、制度論・資源依存論を統合した分析フレーム。
- データモデル: `Organization`, `OrganizationForm`, `Activity`, `Function`, `ImpactRecord`, `Relation`, `Event`, `Source`, `Claim` を核にした出典つき時系列データモデル。
- データセット: オープンデータを中心にした組織・関係・イベント・機能・出典のパイロットデータ。
- 分析パイプライン: 名寄せ、時系列整列、特徴抽出、クラスタリング、ネットワーク分析、系譜推定、評価の再現可能な処理系。
- 公開物: GitHub repo、方法論ノート、データ辞書、可視化、論文草稿、ブログ記事。

原則:

- すべての記述を `Claim` として扱い、出典、確度、解釈者、記録時点を残す。
- 「企業」を標準型にしない。家、宗族、門中、ギルド、修道院、ワクフ、官僚制、大学、系列、DAO、オンライン共同体を同じ比較面に置く。
- 生命メタファーは説明装置ではなく、観察次元を増やすための限定的な比喩として使う。
- 生存を適応の証拠にしない。適応度、正統性、資源アクセス、競争圧は存続結果とは別に測る。
- ボトムアップ分類は既存分類の置換ではなく、既存分類と部分的に重なりつつ新しい構造を発見するものとして評価する。

## 採用する理論的立場 — codex1 の 4 系譜のうちどれを主・補にするか、生命メタファーの罠回避策

主理論は Hannan and Freeman の個体群生態学とする。理由は、創設、死亡、個体群、ニッチ、密度、組織形態を操作化しやすく、長期・多数・比較型データベースの基盤に最も適するためである。ただし、組織内部の意味形成、権力、学習、意図的変革を弱く扱う限界があるため、単独では使わない。

補助理論は 3 系統に分ける。

- Maturana-Luhmann のオートポイエーシス: 境界、決定、記録、儀礼、規則、通信が次の組織作動を生む仕組みを記述するために使う。
- Stafford Beer のサイバネティクス: 分権と統合、現場単位と全体方針、情報遅延、環境応答性を診断語彙として使う。
- 制度論・資源依存論: 効率や適応だけでなく、正統性、模倣、規範、強制、外部資源、権力、交渉を扱う補正項として使う。

Spencer の有機体的社会観は採用理論ではなく、批判対象・系譜上の出発点として扱う。長期史を単線的な分化・統合・進歩物語にしないための警戒標識とする。

生命メタファーの罠回避策:

- Reification 回避: 「組織が生命である」と書かず、「どの作動を生命現象に類比しているか」を明示する。
- 循環論法回避: 存続を適応の証拠にせず、資源条件、正統性条件、競争圧、制度環境を別指標で記録する。
- 境界曖昧性への対応: 法的境界、成員境界、資産境界、意思決定境界、ブランド境界、通信境界を分けて記録する。
- 機能主義の後知恵回避: 制度の存在理由を現在見える機能から逆算せず、権威、正統性、強制、偶然、模倣も説明候補に入れる。
- 生物学概念の過剰輸入回避: 淘汰、進化、ニッチ、代謝、免疫、繁殖、死を使う場合は、社会組織での対応関係を限定する。

## フェーズ計画（v0.0 → v1.0 まで 6〜8 段階） — 各フェーズで 期間 / インプット / アクティビティ / Deliverables / 完了条件 / 主要リスク

### Phase 0: v0.0 Research Synthesis

期間: 2〜4 週間想定。

インプット: `research/A`〜`F`, `codex1`〜`codex3`。

アクティビティ:

- 理論、歴史系譜、分類軸、データ源、方法論、東アジア視点を統合する。
- 未確認・要追跡項目を、文献確認、データ確認、倫理確認に分ける。
- 最小スコープを「オープンデータ + 代表事例 + claim-based schema」に固定する。

Deliverables:

- 統合研究メモ。
- v0.1 のスコープ定義。
- 用語集と暫定分類軸。

完了条件:

- 主理論・補助理論・除外する説明方式が明文化されている。
- v0.1 で入力する組織種別と対象データ源が決まっている。

主要リスク:

- 理論統合が広がりすぎる。
- 生命メタファーが分析概念ではなく宣言文になる。

### Phase 1: v0.1 Minimal Claim-Based Database

期間: 2〜4 ヶ月想定。

インプット: codex2 の `v0.1` モデル、D の利用優先度 Top 10、B/C/codex3 の比較次元。

アクティビティ:

- `organization`, `organization_form`, `organization_form_assignment`, `source`, `claim` を実装する。
- Wikidata, GLEIF, ROR/OpenAlex, gBizINFO/法人番号/EDINET のうち再配布可能な範囲を優先して取り込む。
- 古代・中世・東アジア・現代法人・学術組織・DAO/オンライン共同体から小規模な代表サンプルを作る。

Deliverables:

- v0.1 schema。
- データ辞書。
- 代表組織 200〜500 件程度の仮パイロットデータ。
- 出典・確度・日付精度の入力ルール。

完了条件:

- 全レコードが Source/Claim に紐づく。
- `exact`, `year`, `century`, `period`, `unknown` などの日付精度を扱える。
- 近代法人以外の組織が下位扱いされずに入力できる。

主要リスク:

- 名寄せより先に分類を固定してしまう。
- Wikidata の粒度・欠落を真実として扱ってしまう。

### Phase 2: v0.2 Activity / Function / Impact Separation

期間: 2〜3 ヶ月想定。

インプット: codex2 の `activity`, `function`, `impact_record`、E のイノベーション機能、B/C/codex3 の機能次元。

アクティビティ:

- 活動、機能、インパクトを分離して入力する。
- 機能タイプを `resource_mobilization`, `coordination`, `risk_pooling`, `knowledge_transmission`, `legitimation`, `boundary_maintenance`, `innovation`, `conflict_resolution` などに整理する。
- 東アジア事例では「文化」フィールドに丸めず、継続原理、信用メカニズム、場、名跡、祭祀、制度的保護を機能として分解する。

Deliverables:

- v0.2 schema migration。
- Activity/Function/Impact 入力ガイド。
- 代表組織 50〜100 件の深掘り注釈データ。

完了条件:

- 「何をしたか」と「何の機能を果たしたか」が別フィールドで入力されている。
- 同一活動が複数機能を持つケースを表現できる。

主要リスク:

- 機能ラベルが後知恵の説明になる。
- インパクト評価が価値判断に寄りすぎる。

### Phase 3: v0.3 Relations and Events

期間: 3〜4 ヶ月想定。

インプット: codex2 の `relation`, `event`, `event_organization`, `event_relation`、B の遷移・併存パターン、E の event/trajectory/ecosystem 視点。

アクティビティ:

- 創設、解散、合併、分裂、改称、移転、改革、危機、ガバナンス変更を Event として記録する。
- 支配、同盟、競争、継承、分派、資源交換、資本関係、知識移転、プロトコル依存を Relation として記録する。
- 系譜を tree だけでなく network として扱うため、水平伝播、模倣、制度的同型化をエッジ化する。

Deliverables:

- v0.3 schema migration。
- 主要組織形態の遷移ネットワーク試作。
- 代表ケースの系譜図 3〜5 件。

完了条件:

- 分裂、統合、継承、支配、資源交換を Event と Relation の両方から辿れる。
- Relation の強度が根拠ありの場合のみ記録され、欠損が許容される。

主要リスク:

- 影響関係と類似関係を混同する。
- 系譜図が物語として美しすぎ、根拠の薄いエッジを増やす。

### Phase 4: v0.5 Temporal Facets and Comparative Casebook

期間: 3〜5 ヶ月想定。

インプット: codex2 の temporal facet、B の時代横断指標、C の共通分類次元、codex3 の東アジア次元。

アクティビティ:

- 統治、規模、資源基盤、領域、技術基盤、アイデンティティ、メンバーシップの時系列 facet を実装する。
- 近代以前、東アジア、現代法人、学術組織、プラットフォーム/DAO を含む比較ケースブックを作る。
- 矛盾する史料を Claim の確度と解釈ノートで管理する。

Deliverables:

- v0.5 schema migration。
- 比較ケースブック初版。
- facet 入力済み代表組織 100〜200 件。

完了条件:

- 同一組織の長期変化と別組織への変換を Event で区別できる。
- 期間が重なる facet と矛盾 claim を保持できる。
- 東アジアの「家」「暖簾」「宗族」「会館」「系列」などが法人モデルに圧縮されず記述されている。

主要リスク:

- temporal facet が JSONB の自由記述に流れ、比較可能性を失う。
- 近代法人データだけが厚くなり、歴史・非西洋データが薄いままになる。

### Phase 5: v0.7 Feature Extraction and Bottom-Up Typology

期間: 4〜6 ヶ月想定。

インプット: F の推奨パイプライン、C の分類次元、E のイノベーション指標、v0.5 データ。

アクティビティ:

- 調整メカニズム比率、権威集中度、フォーマライゼーション、環境結合度、価値・意味系、ライフサイクル、法的・経済的器を特徴量化する。
- テキスト埋め込み、ネットワーク特徴、構造化数値を統合し、UMAP/HDBSCAN、BERTopic、階層クラスタリングを試す。
- 内部整合性、外部妥当性、予測妥当性、構成概念妥当性で評価する。

Deliverables:

- feature schema v0.7。
- クラスタリング実験ノート。
- 暫定ボトムアップ分類。
- 評価レポート。

完了条件:

- 少なくとも 3 種類の特徴量セットでクラスタリング結果を比較している。
- 既存分類との ARI/NMI、クラスタ安定性、専門家レビューの準備がある。
- 「既存分類と完全一致」ではなく「部分重複しつつ新規構造を発見」という評価軸で判断している。

主要リスク:

- embedding が公開資料の多い上場大企業に偏る。
- クラスタを発見ではなく命名作業で正当化してしまう。

### Phase 6: v0.9 Genealogy, Network, and Validation

期間: 4〜6 ヶ月想定。

インプット: v0.7 分類、v0.3 relation/event、F の系統推定・ネットワーク分析、E の VSR 接合。

アクティビティ:

- 組織形態の発生、増殖、消滅、分岐、水平伝播、模倣、制度的同型化をネットワークとして描く。
- tree 型の系譜推定と network 型の系譜推定を比較する。
- out-of-time、out-of-domain、専門家 panel、case-based validation を行う。

Deliverables:

- 系譜ネットワーク v0.9。
- 代表系譜図 5〜10 件。
- 妥当性評価レポート。
- 論文骨子。

完了条件:

- 系譜エッジごとに relation type、期間、根拠、確度がある。
- 生命メタファーの対応関係と限界が図注または本文に明記されている。
- 系譜が tree で表せない箇所を network として表現している。

主要リスク:

- 系譜図が因果を過剰に示唆する。
- 非西洋・インフォーマル組織のエッジが資料不足で欠落する。

### Phase 7: v1.0 Public Release and Scholarly Package

期間: 3〜4 ヶ月想定。

インプット: v0.9 データ、評価レポート、論文骨子、ライセンス確認。

アクティビティ:

- 再配布可能データと再配布不可データを分離する。
- GitHub repo、データ辞書、再現スクリプト、可視化、論文草稿、ブログ記事を整える。
- ライセンス、GDPR、Indigenous Data Sovereignty、商用 DB 利用条件を最終確認する。

Deliverables:

- v1.0 release。
- public dataset subset。
- reproducible pipeline。
- paper draft。
- public explainer。

完了条件:

- 公開データがライセンス上再配布可能なソースだけで構成されている。
- 非公開・商用・センシティブデータは再現手順または集計値に限定されている。
- 成功指標の判定結果と限界が明記されている。

主要リスク:

- 公開直前にライセンス違反が見つかる。
- 分類がスティグマ化や政治的誤読を招く。

## 役割分担 — Claude Code, codex CLI, ユーザー（西村優也）の責務

Claude Code:

- 既存ファイル編集、データモデル実装、スクリプト作成、テスト、リファクタリング。
- 受け渡された調査結果を尊重し、勝手に理論・分類を置換しない。
- UI や可視化を作る場合は実ブラウザで確認する。

codex CLI:

- 並列調査、批判的レビュー、ロードマップ化、データ変換、検証観点の整理。
- `research/` と `plan/` の文書統合、スキーマ案、評価基準、リスク登録簿の作成。
- 破壊的操作、PR 作成、外部投稿、商用 DB 利用、公開範囲変更はユーザー確認を取る。

ユーザー（西村優也）:

- 研究目的、公開戦略、理論的立場、対象範囲、倫理判断の最終決定。
- 専門家レビュー、共同研究者、データ利用許諾、商用 DB 契約、論文投稿先の判断。
- 生命メタファーをどの程度前面に出すか、学術成果と公共成果の優先順位を決める。

## ガバナンス — 意思決定のチェックポイント、品質ゲート

意思決定チェックポイント:

- Gate 0: 理論ゲート。主理論・補助理論・避ける理論物語を確定する。
- Gate 1: データゲート。対象データ源、再配布可否、個人情報、データ主権の確認を行う。
- Gate 2: スキーマゲート。法人中心・西洋中心・現代企業中心に偏っていないかを確認する。
- Gate 3: 分類ゲート。クラスタの安定性、既存分類との関係、専門家解釈可能性を確認する。
- Gate 4: 系譜ゲート。エッジの根拠、期間、確度、因果主張の強さを確認する。
- Gate 5: 公開ゲート。ライセンス、GDPR、Indigenous Data Sovereignty、商用 DB 条件、スティグマ化リスクを確認する。

品質ゲート:

- Source/Claim なしの実体記述は v0.1 以降の正式データに入れない。
- 日付は精度を必ず持つ。正確な日付がない場合は `century`, `period`, `unknown` として扱う。
- 既存分類、機能分類、ボトムアップ分類を混同しない。
- 代表ケースには西欧近代企業以外を必ず含める。
- 分類・系譜・評価結果には、限界、欠落、代替解釈を併記する。

| 8 | クラスタの過剰解釈 | 統計的クラスタに本質的な組織種名を付ける | 誤分類・スティグマ化 | 内部、外部、予測、専門家評価を triangulate する | codex CLI、ユーザー |
| 9 | 系譜エッジの因果過剰 | 類似・同時発生を影響関係として描く | 誤った歴史物語になる | relation type と根拠強度を分ける | codex CLI |
| 10 | スキーマ複雑化 | 初期から全概念を実装しようとする | 実装停止、入力不能 | v0.1 から段階追加し、未使用抽象を作らない | Claude Code、codex CLI |

## 倫理・データ・ライセンス — Indigenous data sovereignty, GDPR, 商用 DB ライセンス

オープン公開の第一候補は、Wikidata, GLEIF, ROR, OpenAlex, Crossref, Maddison, Companies House, gBizINFO, EDINET, Charity Commission, USPTO, PatentsView, GovInfo, EU Open Data, UN Data など、商用利用・再配布が可能と整理されたソースに限定する。CC-BY や CC-BY-SA は帰属表示と継承条件を満たす。CC-NC、商用 DB、有料 API、大学ライセンスデータは公開データセットに混ぜない。

GDPR と個人情報保護については、役員、所有者、寄付者、リークデータ、オンチェーン関係など個人を含む関係データを慎重に扱う。公開版は原則として組織単位・集計単位に寄せ、個人名が不可欠な場合は出典、公共性、文脈、最小化を確認する。

Indigenous Data Sovereignty については、D-PLACE, SCCS, eHRAF などの民族誌・人類学データが植民地的収集文脈を含みうることを前提にする。分類の材料として使う場合も、当該社会を固定的な「組織タイプ」に還元せず、記述的引用、出典明示、当事者のデータ主権、利用制限を確認する。

商用 DB については、Compustat, CRSP, Orbis, S&P Capital IQ, PitchBook, CB Insights, MSCI, Sustainalytics, eHRAF, INITIAL/SPEEDA, PATSTAT などを、公開版の元データにしない。利用する場合は、論文内の集計値、再現不能な派生指標、またはアクセス権を持つ研究者向けの再現手順に限定する。

## 成功指標 — 何をもって「ボトムアップ分類が妥当」「系譜が描けた」とするか

ボトムアップ分類が妥当といえる条件:

- 内部整合性: cluster cohesion/separation、HDBSCAN persistence、bootstrap/subsampling stability が確認できる。F の基準を仮置きとして、Silhouette 0.4 以上、Jaccard 0.75 以上は目安に留め、領域ごとに解釈する。
- 外部妥当性: SIC/NAICS、Hannan-Freeman の form/niche、法的形態、既存タイポロジーと ARI/NMI で部分重複しつつ、新しい構造も出ている。
- 予測妥当性: t 期の分類が t+k 期の成長、消滅、合併、分裂、イノベーション、関係形成を偶然より有意に説明する。
- 構成概念妥当性: 専門家レビュー、card-sort、代表ケースの質的読み込みで、クラスタの説明が複数評定者に耐える。
- 倫理・社会的妥当性: 分類がスティグマ化、文化本質主義、弱小組織の不可視化を生まないように説明されている。

系譜が描けたといえる条件:

- 各ノードに source-backed claim があり、時代、地域、組織型、活動、機能、関係、イベントが最低限記録されている。
- 各エッジに relation type、期間、方向、確度、根拠がある。
- 継承、模倣、分裂、統合、支配、資源交換、知識移転、制度的同型化、プロトコル依存を区別できる。
- tree と network の使い分けが明示され、水平伝播やハイブリッド形態が tree に押し込められていない。
- 西洋近代企業だけでなく、家、宗族、門中、ギルド、修道院、ワクフ、大学、官僚制、系列、プラットフォーム、DAO などが同じ説明体系内で比較できる。
- 生命メタファーの対応関係と限界が、本文、図注、データ辞書のいずれかに明記されている。
